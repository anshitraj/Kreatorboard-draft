5. OFFICIAL / OPEN-SOURCE AUTOMATION STRATEGY
==================================================

Do NOT use sketchy scraping as the core architecture.

Use official integrations and open-source tools where possible.

Implement per provider:

A. Gmail
- OAuth connect
- fetch labels, threads, selected metadata, sent mail capability
- collaboration inbox views
- use push notifications / watch architecture where appropriate
- sync only required labels to reduce quota use
- maintain thread mapping in DB
- allow smart lead classification
- never fetch entire mailbox unnecessarily

B. Google Calendar
- OAuth connect
- read events
- create scheduling / availability events
- use watch channels / push style architecture where possible
- sync upcoming windows and booked calls
- show next meetings in dashboard

C. Cal.com
- allow manual booking link mode
- if API credentials exist, sync booking metadata
- otherwise do not fake deep integration
- still support embed and profile CTA

D. Telegram
- support Telegram login / auth-compatible flows where relevant
- support Telegram bot-based integration
- allow creator to connect public channel / group / bot
- store Telegram handles and linked channels
- ingest accessible metadata from bot-authorized surfaces
- support Telegram Mini App compatible surface if useful later
- show real limitations if bot lacks permissions

E. Discord
- use OAuth2 for identity and allowed guild data
- support bot install flow where needed
- support webhook/event-driven sync where appropriate
- map accessible guild/community metadata
- do not overpoll
- respect rate limits
- use cached summaries and deltas

F. X / Twitter
- architect for 3 modes:
  1. BYO API credentials mode
  2. manual CSV import mode
  3. public profile link mode
- do not assume free deep analytics access
- do not make X required for the product to work
- never claim exact X metrics if not actually fetched from authorized sources
- show “limited mode” clearly

G. Wallet
- use WalletConnect / Reown AppKit
- support EVM wallet connection first
- architecture should allow Solana later
- store connected wallet(s)
- support SIWE-style auth or wallet linking if appropriate
- use wallet as identity + payout destination + proof layer

H. Website / blog / portfolio
- allow importing from website metadata, RSS feeds, or provided URLs
- extract title, description, social links, featured content links
- create portfolio blocks automatically
